# タスク管理

タスクは Tsunagi の作業の最小単位です。Claude セッション、git worktree、進捗追跡、エディタ統合——すべての機能はタスクにぶら下がっています。タスクが何であり何を所有しているかを理解すれば、Tsunagi の大部分を理解したことになります。

## タスクとは

Tsunagi のタスクは以下を束ねるレコードです:

| フィールド                  | 意味                                                          |
| --------------------------- | ------------------------------------------------------------- |
| `title`                     | 人間が読む短いタイトル                                        |
| `description`               | Markdown 形式の詳細、計画、PRD など                           |
| `owner` / `repo` / `branch` | git コンテキスト — どのリポジトリのどのブランチに乗っているか |
| `status`                    | ライフサイクル上の位置 (後述)                                 |
| `order`                     | ステータス列内での優先度 (ドラッグで並び替え可)               |
| `effort`                    | おおまかなサイズ見積 (`S` / `M` / `L` / `XL`)                 |
| `tabs`                      | このタスクにぶら下がる 0 個以上の Claude セッション           |
| `worktreeStatus`            | Tsunagi が管理する git worktree の内部状態                    |

各タスクはディスク上に **ちょうど 1 つの git worktree を所有** し、実作業はすべてその worktree 内で行われます。Tsunagi が起動する Claude セッション経由でも、あなた自身の手動操作でも、worktree 内で完結します。

## タスクの作成

タスク一覧パネルで **+ Add Task** をクリックして、以下を入力します:

- **Title** — 何をしてほしいか
- **Repository** — 登録済みのリポジトリから選択
- **Branch** — Tsunagi が worktree を作る git ブランチ。意味のある名前を付けてください。Tsunagi は後からリネームしません
- **Description** — 任意。Claude に読ませる計画や PRD を書くのに便利です
- **Effort** — 任意のサイズヒント

**Create** をクリックすると、Tsunagi は以下を実行します:

1. データベースに `backlog` ステータスの新しいタスク行を挿入
2. `~/.tsunagi/workspaces/<owner>/<repo>/<branch-dir>/` にデフォルトブランチをベースとした git worktree を作成
3. リアルタイムイベントをブロードキャストして、接続中のすべてのブラウザタブに即座にタスクを表示

これだけです——手動の `git` コマンドもシェル操作も不要です。worktree は Claude (またはあなた自身) が作業を始められる状態になっています。

## タスクの編集

タスクカードをクリックすると **タスク詳細** 画面が開きます。そこで以下を編集できます:

- Title と description (Markdown エディタ、ライブプレビュー付き)
- Status (列間のドラッグ、またはステータスセレクタ)
- Effort
- ステータス内での順序 (ドラッグで並び替え)

ほとんどの編集はその場で保存され、別途「保存」ステップは不要です。メインビューのタスク一覧もリアルタイムで更新されます。

`owner` / `repo` / `branch` はタスク作成後は **変更不可** です——これらは worktree を決定するので、変更するには worktree を作り直す必要があります。別のブランチにタスクを移したい場合は、新しいタスクを作成してください。

## タスクの削除

タスクを削除すると以下が削除されます:

- データベースからタスクレコード (論理削除 — 行は `deleted: true` でマークされ UI から非表示)
- ぶら下がるすべての tab セッション (PTY プロセスは kill される)
- ディスク上の git worktree (未コミットの変更を含む)

削除は 3 箇所から可能です:

- **タスク一覧** — 右クリックまたはカードメニュー
- **タスク詳細画面** — オーバーフローメニュー
- **Claude から MCP 経由** — Claude に依頼すれば `tsunagi_delete_task` を呼び出せます ([Tsunagi Plugin](/ja/features/plugin/) を参照)

<Callout type="warning">
  タスクを削除すると、worktree の未コミットの変更も **失われます**。残したい変更は push または stash
  しておいてください。
</Callout>

bare リポジトリと `.default` worktree はタスク削除の影響を **一切受けません** — 該当タスクのブランチディレクトリだけが削除されます。

## ステータスの自動更新

タスクは以下のステータスを通過します:

`backlog` → `planning` → `coding` → `reviewing` → `done`

ステータスはいつでも手動で変更できますが、Tsunagi は **Claude の作業に応じてステータスを自動更新** します。バンドルされた Tsunagi plugin には、いつ `tsunagi_update_task` を呼んでステータスを進めるべきかを Claude に教える小さな skill が含まれています。

| ステータス  | 状態                                       |
| ----------- | ------------------------------------------ |
| `backlog`   | タスクは存在するが、まだ誰も作業していない |
| `planning`  | アプローチについて人間と Claude が対話中   |
| `coding`    | Claude が積極的に実装中                    |
| `reviewing` | 実装完了、人間のレビュー待ち               |
| `done`      | 作業完了、人間が確認済み                   |

これにより、物理的なカンバンボードのようにタスクがリアルタイムで流れていく様子を観察できます——誰もカードを物理的に動かさなくても。

## タブ — 1 タスクで複数 Claude セッション

**タブ** はタスクにぶら下がる 1 つの Claude Code セッションです。1 つのタスクは任意の数のタブを持てて、それぞれが同じ worktree 内で独立に動作します。

  alt="タスク内の複数タブ"

/>

なぜ複数タブか?

- 機能を実装するタブとテストをデバッグするタブを同時に動かす
- メインの会話を失わずに使い捨ての探索的セッションを起動する
- 2 つの異なるアプローチを横並びに比較する

すべてのタブが同じ worktree を共有するので、1 つのタブで書いたファイルは他のタブからも即座に見えます。各タブはフル機能の Claude Code セッションです——ターミナルで `claude` を起動した時と同じもの——なので、Claude Code のあらゆる機能をタブの中で使えます。

### タブの起動方法

タスク詳細画面で **+ New Tab** をクリックすると、Tsunagi は以下を実行します:

1. `node-pty` を使って、タスクの worktree 内で PTY バックエンドのシェルを起動
2. Tsunagi 設定画面で定義された環境変数を注入 ([環境変数管理](/ja/features/environment-variables/) 参照)
3. Claude Code 自身が使うのと同じ認証情報 (`ANTHROPIC_API_KEY` または `CLAUDE_CODE_OAUTH_TOKEN`) でそのシェルで Claude Code を起動
4. Claude の hooks を Tsunagi バックエンドに接続し、UI からセッションの動作をリアルタイムに読めるようにする

### リロードを跨いだ永続化

タブセッションはサーバー側で永続化されています。ブラウザをリロードしたりネットワーク接続が切れたりしても、PTY プロセスは Tsunagi バックエンドで動き続け、再接続すると不在中の出力も含めてターミナルが復元されます。

タブを完全に閉じたい場合はタブの閉じるボタンを使います——これが PTY プロセスを kill し、タスクからタブレコードを削除します。

## プログレスバー — リアルタイム todo 追跡

タスク一覧の各タスクカードには、Claude の作業状況をリアルタイムに反映する **プログレスバー** が表示されます。

仕組み: Claude が `TodoWrite` (Claude 標準ツールの 1 つ) を呼んで todo リストを記録・更新すると、Claude Code の hook が発火します。バンドルされた Tsunagi plugin が新しい todo を Tsunagi バックエンドに POST し、バックエンドが Socket.IO 経由でブラウザにブロードキャストします。プログレスバーは完了数と未完了数から計算・描画されます。

[チュートリアル](/ja/tutorial/start-session/) が Claude に「todo を作成してから...」と指示させるのは、これが理由です——単なるプロンプトのベストプラクティスではなく、プログレスバーを動かすための仕組みです。

設定は一切不要です。Claude が `TodoWrite` を使えば、プログレスバーは自動的に現れます。

## Monaco エディタ統合

Tsunagi には UI 内でファイルを開ける組み込みの **Monaco エディタ** モーダルがあります。Claude (またはあなた) がターミナルの 1 行プロンプトより長い内容を編集する必要がある時に便利です——commit メッセージの執筆、Markdown プロンプトの編集、設定ファイルの微調整など。

### 起動トリガー

Tsunagi は各 Claude セッションのシェル環境で `EDITOR` を小さなヘルパースクリプトに向けて設定します。ツールがデフォルトエディタを開くとき (`git commit`、`git rebase -i`、各種 CLI の `npx prompt` など)、ヘルパーが `vi` のようなターミナルモードのエディタを起動する代わりに **Monaco モーダル** を Tsunagi UI 上で開きます。

ターミナルビューからキーボードショートカット (通常 `Ctrl+G`) で直接 Monaco を開き、カーソル位置のファイルを編集することもできます。

### なぜ Monaco か

- **シンタックスハイライト** と Markdown プレビューが標準装備
- **複数行編集** — ターミナルの折り返しと戦う必要がない
- **Tsunagi UI の中に留まれる** — アプリを切り替える必要がない

モーダル内で通常通り編集し、保存 (`Cmd+S` または `Ctrl+S`) してディスミスします。Claude セッションに制御が戻り、編集済みファイルを参照して続行します。

## フィルタリングと並び替え

タスク一覧パネルは以下をサポートします:

- リポジトリ・ステータス・フリーテキスト検索での **フィルタリング**
- ステータス列内でのドラッグによるタスクの **並び替え**
- 列ヘッダーメニューからの `done` タスクの **一括削除**

並び替えはリロードを跨いで永続化されます——`order` フィールドはデータベースに保存されます。この仕組みは、あなたが (または Claude が `tsunagi-task-curator` agent 経由で) backlog をまとめて並べ替える時に MCP server が使う仕組みと同じです。

## ディスク上のタスク

内部的には、各タスクは Tsunagi のワークスペースディレクトリ内の **git worktree** に裏打ちされています:

```
~/.tsunagi/workspaces/<owner>/<repo>/
├── .bare/                    # 全 worktree で共有される bare clone
├── .default/                 # デフォルトブランチの読み取り専用リファレンスチェックアウト
├── feat-some-task/           # タスクブランチごとに 1 つのディレクトリ
└── fix-another-task/
```

これにより:

- **タスクの worktree を自分のエディタで直接開ける** (`code ~/.tsunagi/workspaces/owner/repo/feat-xxx/`)。Tsunagi の外で作業しても、Tsunagi UI は Claude セッションを通じてファイル変更を追跡し続けます
- **1 タスクの複数 Claude セッションは 1 つの worktree を共有** するので、お互いの編集が見えます
- **異なるタスクは完全に別々のディレクトリに住む** ので、並列タスクがファイルシステム上で衝突することはありません (bare clone を共有していても)

Tsunagi はこれらの worktree を自動管理します——タスク作成時の `git worktree add`、削除時の `git worktree remove` ——なので、通常は `git worktree` コマンドを自分で実行する必要はありません。
